# 第0章：读懂这份文档所需的基础知识

> **写给完全的小白**：本章不假设你有任何数字电路、计算机体系结构或 Verilog 编程经验。我们会用生活中的类比，一步步建立起理解 e203 处理器所需的直觉。

---

## 0.1 CPU 是什么？

### 从计算器到处理器

想象你在做一道数学题：

```
算步骤：
  1. 打开书，找到题目 43 + 27
  2. 看懂题目：这是一道加法题，两个数是 43 和 27
  3. 用脑子或草稿纸算：43 + 27 = 70
  4. 把 70 写到答题纸上
  5. 翻到下一道题，重复上述步骤
```

CPU（中央处理器）做的事情几乎一样：

| 你做的 | CPU做的对应操作 |
|--------|----------------|
| 打开书，找到题目 | **取指（IFU）**：从内存读出下一条指令 |
| 看懂题目是加法 | **译码（Decode）**：解析指令的操作类型 |
| 算出结果 | **执行（Execute）**：ALU 进行计算 |
| 写到答题纸 | **写回（Write-back）**：结果写入寄存器 |
| 翻到下一题 | **PC+4**：程序计数器指向下一条指令 |

这个循环——**取指 → 译码 → 执行 → 写回** ——就是 CPU 的核心工作。

### 一条"指令"是什么？

指令就是用数字编码的操作命令。例如，"把寄存器 x1 和 x2 的值相加，结果放进 x3"，在 RISC-V 架构里被编码成一个 32 位（bit）的数字：

```
0000000_00010_00001_000_00011_0110011
│       │     │    │   │     └── 操作类型（R型指令）
│       │     │    │   └──────── 目标寄存器 x3
│       │     │    └──────────── 功能码（加法）
│       │     └─────────────────  源寄存器1 x1
│       └───────────────────────  源寄存器2 x2
└───────────────────────────────  功能码高位（普通加法，非减法）
```

每条指令都是一个 32 位的数字，CPU 读进来、解析、执行，如此往复，程序就运行了起来。

### 什么是"寄存器"？

寄存器就是 CPU **内部的草稿纸**，速度极快，但数量极少。

e203 有 **32 个通用寄存器**，命名为 `x0` 到 `x31`：

```
+-----+------+-----------------------------------------------+
| 编号 | 名称  | 约定用途                                       |
+-----+------+-----------------------------------------------+
| x0  | zero | 永远是 0，写入无效                             |
| x1  | ra   | 返回地址（函数调用用）                          |
| x2  | sp   | 栈指针                                         |
| x3  | gp   | 全局指针                                       |
| x4  | tp   | 线程指针                                       |
| x5-7 | t0-2 | 临时变量                                      |
| x8  | s0/fp| 保存寄存器/帧指针                              |
| x9  | s1   | 保存寄存器                                     |
| x10-11 | a0-1 | 函数参数/返回值                           |
| x12-17 | a2-7 | 函数参数                                  |
| x18-27 | s2-11| 保存寄存器                                |
| x28-31 | t3-6 | 临时变量                                  |
+-----+------+-----------------------------------------------+
```

## 0.2 什么是 RISC-V 指令集？

### 指令集就是 CPU 和程序的"语言协议"

就像手机在出厂前，厂商定好了"发出这个音调就是开机"、"按这个组合键就是截图"一样，**指令集（ISA, Instruction Set Architecture）** 就是 CPU 和软件之间的约定：

- 软件说："我用 `0110011_...` 这个二进制数来表示加法"
- CPU 说："好，我看到这个数字就执行加法"

**RISC-V** 是一套 **开源免费** 的指令集标准（*Reduced Instruction Set Computer V*）。它和 ARM、x86 一样是指令集，但：

- **开放**：任何人、任何公司都可以免费使用
- **简洁**：基础指令只有 47 条（RV32I 基础整数集）
- **模块化**：可以按需添加扩展（M=乘除、A=原子操作、C=压缩指令...）

e203 实现的是 **RV32IMAC**：
```
RV  = RISC-V
32  = 32位地址空间
I   = 基础整数指令集
M   = 乘除法扩展（MULDIV）
A   = 原子操作扩展（AMO）
C   = 压缩指令扩展（16位短指令，节省代码存储空间）
```

### 什么是"压缩指令（RVC）"？

标准 RISC-V 指令是 32 位宽。RVC（Compressed）提供了功能相同但只需 16 位的"短版指令"，节省一半内存。

e203 同时支持 32 位和 16 位指令，这给取指单元带来了处理"拼接跨界"的额外复杂度（第2章会详细讲）。

## 0.3 什么是 Verilog / RTL？

### 数字电路 vs 软件程序

软件程序描述的是**时间顺序**：先做这步，再做那步。

硬件电路描述的是**空间结构**：这个触发器连接那个组合逻辑，那个组合逻辑又连接另一个触发器。

**RTL（Register Transfer Level，寄存器传输级）** 是一种抽象层次，描述数据在**寄存器（触发器 D Flip-Flop）** 之间如何流动和变换。

**Verilog** 是用来描述这个结构的编程语言（硬件描述语言 HDL）。

### 三个最关键的概念

#### 1. D 触发器（D Flip-Flop，DFF）——电路的记忆单元

```
       ┌──────┐
  D ──►│      │──► Q（输出）
       │  DFF │
CLK ──►│      │
  ↑    └──────┘
时钟上升沿时，将 D 的值"拍"进去，Q 保持直到下一个上升沿
```

e203 中用到的基础 DFF 模块（来自 `sirv_gnrl_dffs.v`）：

```verilog
module sirv_gnrl_dfflr # (parameter DW = 32) (
  input               lden,   // 使能信号：1 才会采样
  input      [DW-1:0] dnxt,   // 下一个值（D端）
  output     [DW-1:0] qout,   // 当前值（Q端，"记住"的值）
  input               clk,    // 时钟
  input               rst_n   // 复位（低有效）
);

always @(posedge clk or negedge rst_n) begin
  if (rst_n == 1'b0)         // 复位时
    qout_r <= 0;             //   清零
  else if (lden == 1'b1)     // 使能时
    qout_r <= dnxt;          //   记住新值
  // lden=0 时，qout_r 保持不变
end
```

**类比**：想象一个带"拍照"按钮的快照相机：
- `clk` 上升沿 = 按下快门
- `lden=1` = 相机盖打开（允许拍照）
- `lden=0` = 相机盖关上（快门按下也不拍）
- `qout` = 当前照片

#### 2. 组合逻辑——纯粹的"函数计算"

组合逻辑没有记忆，输入变化输出立刻变化，就像数学函数 `y = f(x)`：

```verilog
assign sum = a + b;  // a 或 b 变化时，sum 立刻更新
```

数字电路就是 **组合逻辑 + 触发器** 的结合：组合逻辑做计算，触发器在时钟边沿"存档"结果。

#### 3. 握手协议（Valid-Ready Handshake）——模块间的通信

模块之间传递数据时，需要协商：

```
发送方：valid = 1  →  "我有数据，可以发了"
接收方：ready = 1  →  "我准备好接收了"
只有 valid AND ready 同时为 1 时，数据才真正传递！
```

```
时间轴：  Cycle1   Cycle2   Cycle3   Cycle4
valid  :   1        1        1        1
ready  :   0        0        1        1
                             ↑        ↑
                          第一次    第二次
                          握手成功  握手成功
```

这是 e203 内部所有模块间通信的基础规则，包括 ICB 总线（第1章详解）。

### Verilog 代码阅读快速上手

```verilog
// 模块声明（类似函数定义）
module 模块名 (
  input  信号名,          // 输入信号
  output 信号名,          // 输出信号
  input  [31:0] 数据信号  // 32位宽的信号（[高位:低位]）
);

// 寄存器（触发器，有记忆）
reg [31:0] 某个寄存器;

// 线（组合逻辑，无记忆）
wire [31:0] 某条连线;

// 连续赋值（组合逻辑）
assign 某条连线 = 信号A & 信号B;   // 按位与
assign 输出     = 条件 ? 值A : 值B; // 三元选择

// 时序逻辑（触发器）
always @(posedge clk or negedge rst_n) begin
  if (!rst_n) 寄存器 <= 0;        // 复位
  else if (使能) 寄存器 <= 新值;   // 更新
end

// 模块实例化（调用子模块）
子模块名 #(.参数(值)) 实例名 (
  .端口名(连线名),
  ...
);
```

## 0.4 e203 的定位与特色

### 蜂鸟处理器是什么？

e203 是由**芯来科技（Nuclei System Technology）** 设计的一款开源 RISC-V 处理器核，绰号"蜂鸟"（HBird，HummingBird）。

```
定位：
  超低功耗  ┬─ MCU（微控制器）级别
  嵌入式    └─ 类似 ARM Cortex-M3/M4 的竞品

目标场景：
  • IoT 传感器节点
  • 工业控制器  
  • 可穿戴设备
  • 教学科研（开源可研读）
```

### e203 的三大设计哲学

#### 1. 极简流水线（2级）

绝大多数处理器有 5 级甚至更深的流水线（ARM M4 有 3 级，Cortex-A72 有 15 级）。e203 只设计了 **2级**：

```
                取指                    执行
           ┌──────────┐           ┌──────────────┐
   PC ──►  │  IFU     │ ──指令──► │  EXU         │ ──结果──► 寄存器
           │ (取指单元) │           │ (执行单元)    │
           └──────────┘           └──────────────┘
             第1级                    第2级

（LSU 访存是 EXU 内的长周期子流水线）
```

流水线越深，面积越大，功耗越高。2 级流水线换取的是：
- **极低面积**：适合嵌入式场景
- **极低功耗**：蜂鸟的关键优势
- **简单控制逻辑**：代码可读性更好（适合学习）

#### 2. TCM（紧耦合存储器）架构

不使用 Cache（缓存），而是使用延迟固定的 **TCM（Tightly Coupled Memory）**：

```
带 Cache 的处理器：
  hit:  1-2拍（Cache命中）
  miss: 10-100拍（去DDR取数据）  →  延迟不确定！

e203 用 TCM：
  访问 ITCM/DTCM：固定 1拍   →  延迟完全确定！
```

这对**实时控制**场景非常重要——你需要在确定的时间内响应中断。

#### 3. OITF 解决长指令的依赖问题

乘除法、访存等指令不能在 1 拍内完成。e203 用 **OITF（Outstanding Instruction Track FIFO）** 追踪这些"在飞行中的指令"，保证顺序语义正确（第3章详解）。

## 0.5 快速建立整体印象

在深入每个模块之前，脑海中先刻下这张图：

```
程序运行流程（对应 e203 的各个模块）：

  程序员写的 C 代码：
      int a = 5, b = 3;
      int c = a + b;

  编译后变成 RISC-V 指令（存放在 ITCM 里）：
      0x80000000: addi x10, x0, 5   // x10 = 0 + 5 = 5
      0x80000004: addi x11, x0, 3   // x11 = 0 + 3 = 3
      0x80000008: add  x12, x10, x11 // x12 = x10 + x11

  CPU 执行过程：
  ┌─────────────────────────────────────────────────────────┐
  │  IFU（取指单元）                                          │
  │    PC=0x80000000  ──读ITCM──►  拿到 addi x10,x0,5 指令  │
  │    PC+4 = 0x80000004（顺序执行）                         │
  └─────────────────────────────────────────────────────────┘
                       │ 指令送到 EXU
  ┌─────────────────────────────────────────────────────────┐
  │  EXU（执行单元）                                          │
  │    decode: 识别为 ADDI 指令                               │
  │    regfile: 读 x0 = 0                                    │
  │    ALU: 0 + 5 = 5                                        │
  │    wbck: 写回 x10 = 5                                    │
  └─────────────────────────────────────────────────────────┘
```

## 本章小结

| 概念 | 一句话解释 | e203 中对应 |
|------|-----------|------------|
| CPU | 反复执行"取指-执行"循环的机器 | e203 整体 |
| 指令 | 用数字编码的操作命令（32位） | RISC-V RV32IMAC |
| 寄存器 | CPU内部的草稿纸（32个，极快） | `e203_exu_regfile` |
| 流水线 | 多步骤并行以提高吞吐量 | 2级（IFU+EXU） |
| D触发器 | 在时钟边沿存储1bit信息的单元 | `sirv_gnrl_dffs.v` |
| 握手协议 | valid+ready同时为1才传数据 | 所有模块间通信 |
| TCM | 延迟固定的紧耦合存储器 | ITCM + DTCM |
| OITF | 追踪"飞行中"长指令的队列 | `e203_exu_oitf` |
