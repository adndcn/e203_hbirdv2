# 第2章：取指单元（IFU）

> **本章目标**：理解 CPU 如何"读书"。从 PC（程序计数器）到 IR（指令寄存器），一条指令是怎样被取出来的——包括如何选择读哪个地址、如何处理 64bit 数据里切出 32bit 指令、以及如何"猜"下一条指令在哪。

---

## 2.1 IFU 的整体职责

IFU（Instruction Fetch Unit，取指单元）就是 CPU 的"阅读器"，专门负责从存储器里读出指令。

它的主要任务链：

```
  ┌────────────────────────────────────────────────────────────────┐
  │                    IFU (e203_ifu.v)                            │
  │                                                                │
  │  ①  ifetch.v           ②  ift2icb.v         ③  minidec.v     │
  │  ─────────────         ──────────────        ──────────────   │
  │  • 维护 PC              • 把取指请求         • 对当前指令      │
  │  • 决定取哪个地址        翻译成 ICB 总线       做"浅层译码"    │
  │  • 管理 IR 寄存器        命令发给 ITCM         判断是否是      │
  │  • 遇到跳转时更新 PC     • 处理 64bit→32bit    跳转指令        │
  │                          切割和跨界拼接                        │
  │                                         \                     │
  │  ④  litebpu.v                            输出给 EXU            │
  │  ─────────────                                                 │
  │  • 分支预测             ifu_o_ir   ─────────────────────────►  │
  │  • 提前猜测 PC 跳向哪里  ifu_o_pc                              │
  └────────────────────────────────────────────────────────────────┘
```

它的四个子模块：

| 子模块 | 文件 | 功能一句话 |
|--------|------|-----------|
| `e203_ifu_ifetch` | `e203_ifu_ifetch.v` | IFU 主控：管理 PC 和 IR，驱动整个取指流程 |
| `e203_ifu_ift2icb` | `e203_ifu_ift2icb.v` | 协议转换：取指请求 → ICB 总线命令 |
| `e203_ifu_minidec` | `e203_ifu_minidec.v` | 预译码：识别指令类型（仅在 IFU 内使用） |
| `e203_ifu_litebpu` | `e203_ifu_litebpu.v` | 分支预测：猜测下一个 PC |

## 2.2 程序计数器（PC）：CPU 的书签

### PC 是什么？

PC（Program Counter，程序计数器）就是 CPU 的"书签"，记录着**当前正在执行的指令地址**。

正常情况下，每取一条 32bit 指令，PC 自动加 4（因为一条指令占 4 个字节）：

```
PC = 0x80000000  →  取指令  →  PC = 0x80000004
PC = 0x80000004  →  取指令  →  PC = 0x80000008
PC = 0x80000008  →  取指令  →  PC = 0x8000000C
...
```

但以下情况会打断这个规律：

```
情况1：跳转指令（JAL/JALR/Branch）
  → PC 跳到 目标地址

情况2：异常/中断发生
  → PC 跳到 异常处理入口（mtvec 寄存器的值）

情况3：从异常处理返回（mret 指令）
  → PC 恢复到 mepc 寄存器保存的地址

情况4：调试模式
  → PC 跳到 debug ROM 的入口
```

### PC 存储在哪？

PC 存储在一个 32 位的 D 触发器链（寄存器）里。在 `e203_ifu_ifetch.v` 中：

```verilog
// pc_r 是当前 PC 寄存器
// pc_nxt 是下一拍的 PC（组合逻辑计算）
sirv_gnrl_dfflrs #(`E203_PC_SIZE) pc_dfflrs(
  .lden  (pc_ena),   // 使能：每次取指成功时更新 PC
  .dnxt  (pc_nxt),   // 下一个 PC 值
  .qout  (pc_r ),    // 当前 PC 值
  .clk   (clk  ),
  .rst_n (rst_n)
);
```

## 2.3 IR 寄存器：指令的"暂存器"

取回的指令不会立刻被执行，而是先存入 **IR（Instruction Register，指令寄存器）**，等待 EXU 处理。

IR 是 IFU 和 EXU 之间的"缓冲盒"：

```
ITCM ─── 取指 ───► IR（指令寄存器）─────► EXU 执行
                        │
                    暂时"停在这里"
                    等 EXU 准备好
```

IR 有一个重要设计细节：**分上下两部分单独保存**：

```verilog
// IR 分为高 16bit 和低 16bit 两部分
// 低 16bit 每次都更新（因为压缩指令只有 16bit）
// 高 16bit 只有 32bit 指令时才更新（节省功耗：减少不必要的翻转）

sirv_gnrl_dfflrs #(16) ir_lo_dfflrs(  // 低 16bit，总是更新
  .lden(ir_lo_ena), ...
);
sirv_gnrl_dfflrs #(16) ir_hi_dfflrs(  // 高 16bit，只有 32bit 指令时更新
  .lden(ir_hi_ena), ...
);
```

**为什么这样设计？** 每次 DFF 翻转（0→1 或 1→0）都会消耗动态功耗。既然 16bit 压缩指令用不到高 16bit，就不让它翻转——这是 e203 极致低功耗设计思路的缩影。

## 2.4 ift2icb：取指请求转 ICB 总线命令

`ift2icb`（IFetch To ICB）的名字就说明了它的功能：把取指请求（`ifu_req_*`）翻译成 ICB 总线命令（`ifu2itcm_icb_cmd_*`），发给 ITCM。

### 取指 vs ICB 的区别

```
取指接口（IFU内部）          ICB 总线接口（ITCM侧）
────────────────────         ──────────────────────
ifu_req_valid                ifu2itcm_icb_cmd_valid
ifu_req_ready                ifu2itcm_icb_cmd_ready
ifu_req_pc [31:0]            ifu2itcm_icb_cmd_addr [15:0]
                               （只用低16位，因为ITCM只有64KB）
ifu_rsp_valid                ifu2itcm_icb_rsp_valid
ifu_rsp_instr [31:0]         ifu2itcm_icb_rsp_rdata [63:0]
                               （ITCM返回64bit！需要切割）
```

### 地址路由：去 ITCM 还是去系统内存？

当 PC 发出取指请求时，需要判断目标地址落在哪个范围：

```verilog
// 地址的高位是否匹配 ITCM 的基址高位？
wire ifu_req_pc2itcm = 
  (ifu_req_pc[`E203_ITCM_BASE_REGION] 
      == itcm_region_indic[`E203_ITCM_BASE_REGION]);

// 不在 ITCM 范围内，就去系统内存
wire ifu_req_pc2mem = ~ifu_req_pc2itcm;
```

## 2.5 关键难点：64bit ITCM 到 32bit 指令的切割

### 为什么 ITCM 是 64bit 宽？

ITCM 提供 **64bit（8字节）数据线**，而 RISC-V 指令最多 32bit（4字节）。之所以 64bit：

- 一次可以读出两条 32bit 指令（或四条 16bit 压缩指令）
- 减少 SRAM 访问次数，提高取指效率
- 支持流水线提前预取

### 如何从 64bit 中取出一条 32bit 指令？

PC 地址的低 3 位（`pc[2:0]`）决定了指令在 64bit 数据中的位置：

```
64bit 数据（来自 ITCM）:
┌──────────┬──────────┬──────────┬──────────┐
│ bit63:48 │ bit47:32 │ bit31:16 │ bit15:0  │
│ 第4个16B │ 第3个16B │ 第2个16B │ 第1个16B │
└──────────┴──────────┴──────────┴──────────┘

PC[2:1] = 00   → 取 bit[31:0]   （第1、2个16bit，4字节对齐）
PC[2:1] = 01   → 取 bit[47:16]  （跨第2、3个16bit）
PC[2:1] = 10   → 取 bit[63:32]  （第3、4个16bit）
PC[2:1] = 11   → 跨越64bit边界！（需要两次访问）
```

### 跨 64bit 边界的"Leftover"机制

当 PC 恰好指向 64bit 边界的最后 16bit 时（`PC[2:1] == 2'b11`）：

```
第一次取指（读当前 64bit）：
  ITCM 返回数据：[63:0]
  我们只能得到 [63:48]（低 16bit 的指令）
  → 把 [63:48] 存入 leftover_r 寄存器（16bit 缓存）

第二次取指（读下一个 64bit）：
  ITCM 返回数据：[63:0]（下一个 64bit 块）
  我们取 [15:0]（高 16bit 的指令）
  → 把 leftover_r 和 [15:0] 拼起来 = 完整的 32bit 指令！
```

图示如下：

```
地址:      ...C    ...D    ...E    ...F | ...10   ...11  ...12  ...13
          ┌───────┬───────┬───────┬───────┤───────┬───────┬───────┬───────┐
64bit#1:  │       │       │       │ 指令低│         （下一个 64bit）        │
          └───────┴───────┴───────┴───────┘
          PC[2:1]= 00      01      10  11↑
                                       指令高 16bit 在下一个 64bit 的 [15:0]
                           leftover_r ─┐
                                       ▼
                           [指令低16] + [指令高16] = 完整32bit指令
```

**为什么需要这个机制？** 因为 16bit 的 RVC 压缩指令可以出现在任何 2 字节对齐的位置，所以 32bit 指令完全可能横跨两个 64bit 边界。

## 2.6 minidec：轻量级预译码

预译码（`e203_ifu_minidec.v`）在指令进入 IR 之前，先做一次"粗识别"，目的是：

1. **识别指令长度**：是 16bit（RVC）还是 32bit？
   - 规则：`instr[1:0] != 2'b11` → 16bit；否则 32bit
2. **识别跳转指令**：是否是 JAL/JALR/Branch？（给分支预测用）
3. **读出依赖寄存器索引**：rs1/rs2 的编号（给 OITF 依赖检测用）

```verilog
// minidec 直接复用了 exu_decode 模块
// 只是把大多数输出端口悬空（不需要的部分不接线）
e203_exu_decode u_e203_exu_decode(
  .i_instr(instr),
  ...
  .dec_rv32(dec_rv32),   // 1 = 32bit指令
  .dec_jal (dec_jal),    // 是否是 JAL
  .dec_jalr(dec_jalr),   // 是否是 JALR
  .dec_bxx (dec_bxx),    // 是否是条件分支（Bxx）
  .dec_bjp_imm(dec_bjp_imm),  // 跳转偏移量
  ...
);
```

**设计技巧**：复用 `e203_exu_decode` 而不是单独写一个预译码模块，避免了代码冗余，保证了预译码逻辑和完整译码逻辑的一致性。

## 2.7 liteBPU：轻量级分支预测

### 为什么需要分支预测？

正常情况下，IFU 顺序取指：PC, PC+4, PC+8 ...

但当遇到跳转指令（JAL、JALR、Branch）时，PC 要跳到目标地址。如果等 EXU 算出目标地址再告诉 IFU，IFU 就白白等了一拍——这叫"分支惩罚"。

分支预测就是在 IFU 阶段**提前猜**跳转目标，如果猜对了就没有损失；猜错了再由 EXU 纠正（清流水线，重新取指）。

### liteBPU 的预测策略（极简静态预测）

e203 采用了最简单的**静态预测**策略：

```
┌────────────────────────────────────────────────────────────────┐
│ 指令类型           │ 预测策略                 │ 跳转目标来源    │
├────────────────────┼──────────────────────────┼────────────────┤
│ JAL                │ 一定跳（无条件跳转）     │ PC + 立即数偏移 │
├────────────────────┼──────────────────────────┼────────────────┤
│ JALR  (rs1 = x0)  │ 一定跳（无条件跳转）     │ 0 + 立即数     │
├────────────────────┼──────────────────────────┼────────────────┤
│ JALR  (rs1 = x1)  │ 一定跳                   │ 直接读寄存器x1  │
│                    │ 但需检查x1是否有数据冒险  │ (x1常用作返回地址) │
├────────────────────┼──────────────────────────┼────────────────┤
│ JALR  (rs1 = xN)  │ 一定跳，但需等 EXU 算出  │ 等 EXU         │
│                    │ xN 的值（损失1拍）       │                │
├────────────────────┼──────────────────────────┼────────────────┤
│ Bxx (向后跳，      │ 预测"跳"                │ PC + 立即数偏移 │
│ imm < 0)          │                          │                │
├────────────────────┼──────────────────────────┼────────────────┤
│ Bxx (向前跳，      │ 预测"不跳"              │ PC + 4         │
│ imm > 0)          │                          │                │
└────────────────────┴──────────────────────────┴────────────────┘
```

"向后跳预测为跳"的直觉：大部分循环都是向后跳（循环体结束，跳回循环开头），这样预测正确率比较高。

### 关键：JALR x1 的依赖检测

`x1` 是返回地址寄存器（`ra`），函数返回时会用 `jalr x0, x1, 0`（跳到 x1 所存的地址）。BPU 直接读 x1 可以在 IFU 阶段就知道跳转目标，但前提是 x1 没有被正在执行的指令修改：

```verilog
// jalr_rs1x1_dep：x1 有数据依赖——需要等待
wire jalr_rs1x1_dep = dec_i_valid & dec_jalr & dec_jalr_rs1x1 
    & ((~oitf_empty)               // OITF 非空：有长指令可能正在写 x1
    | (jalr_rs1idx_cam_irrdidx));  // IR 中的指令正在写 x1

// 如果 x1 有依赖，BPU 就等（bpu_wait = 1）
// IFU 停止取下一条指令，直到依赖清除
assign bpu_wait = jalr_rs1x1_dep | ...;
```

这是一个典型的**硬件数据冒险检测（Hazard Detection）**案例。

### 预测信号输出

```verilog
// prdt_taken：预测是否跳转
// JAL/JALR 一定跳；Bxx 依据 imm 的符号位判断
assign prdt_taken = (dec_jal | dec_jalr | (dec_bxx & dec_bjp_imm[`E203_XLEN-1]));

// prdt_pc_add_op1/op2：目标 PC = op1 + op2
// 这两个加数送给 ifetch 中的 PC 加法器直接计算目标地址
```

## 2.8 信号追踪：从 PC 到 IR 的完整数据流

下面以"顺序取一条 32bit 指令"为例，追踪所有经过的信号：

```
Step 1: ifetch 发出取指请求
──────────────────────────────────────────────────────────────
ifu_req_valid   = 1        // 我有取指请求
ifu_req_pc      = 0x80000008  // 取这个地址的指令
ifu_req_seq     = 1        // 这是顺序取指（PC = 上次PC + 4）
ifu_req_seq_rv32= 1        // 上一条是 32bit 指令

Step 2: ift2icb 把请求转发给 ITCM
──────────────────────────────────────────────────────────────
ifu2itcm_icb_cmd_valid = 1
ifu2itcm_icb_cmd_addr  = 0x0008  // 只用低16位（ITCM内的偏移地址）
                                   // = 0x80000008 & 0xFFFF = 0x0008

Step 3: ITCM 返回 64bit 数据
──────────────────────────────────────────────────────────────
ifu2itcm_icb_rsp_valid = 1
ifu2itcm_icb_rsp_rdata = 64'hXXXXXXXX_00208133
                               ↑高32bit（下一条指令）
                                         ↑低32bit（目标指令 add x3,x1,x2）

Step 4: ift2icb 切出 32bit 指令
──────────────────────────────────────────────────────────────
PC[2:1] = 2'b00 → 取低32bit
i_ifu_rsp_instr = 32'h00208133  // add x3, x1, x2

Step 5: ifetch 将指令写入 IR
──────────────────────────────────────────────────────────────
ir_r[31:0] = 32'h00208133       // IR 更新
pc_r       = 0x80000008         // 当前 PC 也锁存

Step 6: 输出给 EXU
──────────────────────────────────────────────────────────────
ifu_o_valid = 1                 // 有效指令准备好了
ifu_o_ir    = 32'h00208133      // 指令内容
ifu_o_pc    = 0x80000008        // 指令的 PC
ifu_o_prdt_taken = 0            // 预测不跳转

Step 7: 同时，PC 更新为下一条
──────────────────────────────────────────────────────────────
pc_nxt = 0x8000000C  // PC + 4（下一条 32bit 指令地址）
```

## 2.9 "响应绕过缓冲"（bypbuf）：防死锁的小机制

ift2icb 里有一个不起眼但很重要的小模块：`bypass buffer`（绕过缓冲）。

**为什么需要它？** 想象这种情况：

```
1. IFU 从 ITCM 取指，ITCM 返回数据
2. IFU 想把指令送给 EXU，但 EXU 的 ready = 0（EXU 忙）
3. IFU 等待... 但 IFU 占着 ITCM 的响应通道不放
4. LSU 也要访问 ITCM（存数据），但 ITCM 被 IFU 阻塞了
5. EXU 正在等 LSU 的响应（访存结果）...
6. 死锁！
```

`bypbuf` 是一个深度为 1 的 FIFO：当 EXU 暂时没空接收时，能把 IFU 已经取回的指令临时存一下，让 ITCM 可以释放去响应 LSU，从而打破死锁。

```verilog
sirv_gnrl_bypbuf #(.DP(1), .DW(`E203_INSTR_SIZE+1)) 
  u_e203_ifetch_rsp_bypbuf(
    .i_vld(i_ifu_rsp_valid),
    .i_rdy(i_ifu_rsp_ready),
    .o_vld(ifu_rsp_valid),
    .o_rdy(ifu_rsp_ready),
    .i_dat(ifu_rsp_bypbuf_i_data),  // {err, instr}
    .o_dat(ifu_rsp_bypbuf_o_data),
    ...
  );
```

## 本章小结

| 机制 | 要解决的问题 | 实现方式 |
|------|------------|---------|
| PC 寄存器 | 记住当前执行到哪 | DFF，跳转/异常时更新 |
| IR 双半分离 | 节省压缩指令的动态功耗 | 高16bit单独使能 |
| ift2icb | ITCM 是64bit但指令是32bit | PC[2:1]选择切割窗口 |
| leftover 缓存 | 指令跨越64bit边界 | 16bit寄存器暂存前半截 |
| minidec | BPU和OITF需要提前知道指令类型 | 复用 exu_decode，不接大部分输出 |
| liteBPU | 避免每次跳转都等EXU算结果 | 静态规则+直接读x1 |
| bypbuf | IFU阻塞时防止ITCM死锁 | 深度1的旁路FIFO |
