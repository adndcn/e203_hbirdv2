# 第1章：整体架构鸟瞰

> **本章目标**：建立 e203 的"地图感"。看懂模块树（SoC 有几层、各层做什么）、地址空间分布（谁住在哪个地址）、以及模块间通信的标准语言——ICB 总线协议。

---

## 1.1 从芯片顶层往下看：六层模块树

e203 的完整系统（称为 HBirdv2 SoC）是一棵层次分明的模块树。从最外层的芯片引脚，一路往下到最核心的 ALU，共有六层：

```
第1层  e203_soc_top            ← 芯片顶层，对外的PAD引脚在这里
  │
  第2层  e203_subsys_top        ← 子系统顶层，划分主域和低功耗域
  │  │
  │  ├─ 第3层  e203_subsys_main ← 主处理系统（正常工作时运行的部分）
  │  │    │
  │  │    ├─ e203_subsys_hclkgen  时钟生成（PLL）
  │  │    ├─ e203_cpu_top ──────── ★ CPU核+存储器，本文档的主角
  │  │    ├─ e203_subsys_plic      平台级中断控制器
  │  │    ├─ e203_subsys_clint     软中断与计时器
  │  │    ├─ e203_subsys_perips    外设（UART/SPI/GPIO）
  │  │    └─ e203_subsys_mems      片上ROM（启动代码）
  │  │
  │  └─ 第3层  sirv_aon_top    ← AON低功耗域（Always-On，芯片睡眠时也运行）
  │         └─ RTC / 看门狗 / PMU
  │
  ├─ debug/  调试模块（JTAG）
  ├─ fab/    总线互联（ICB路由）
  ├─ general/ 通用基础库
  ├─ mems/   片上ROM
  └─ perips/ 外设库
```

### 重点深挖：`e203_cpu_top` 的内部结构

`e203_cpu_top` 是文档的核心主角，它由两部分组成：

```
e203_cpu_top
  ├─ e203_cpu          ← "大脑"——处理器核
  │    ├─ e203_core    ← 流水线顶层（IFU + EXU + LSU + BIU）
  │    │    ├─ e203_ifu   取指单元
  │    │    ├─ e203_exu   执行单元
  │    │    ├─ e203_lsu   访存单元
  │    │    └─ e203_biu   总线接口单元
  │    │
  │    ├─ e203_itcm_ctrl  ITCM控制器（指令紧耦合存储器接口）
  │    ├─ e203_dtcm_ctrl  DTCM控制器（数据紧耦合存储器接口）
  │    ├─ e203_clk_ctrl   时钟门控管理
  │    ├─ e203_reset_ctrl 复位序列管理
  │    └─ e203_irq_sync   中断信号同步
  │
  └─ e203_srams         ← "记忆"——物理SRAM
       ├─ e203_itcm_ram  ITCM SRAM（64位宽）
       └─ e203_dtcm_ram  DTCM SRAM（32位宽）
```

## 1.2 两级流水线：e203 的核心结构

e203 整个处理过程可以形象地理解为"工厂的两条流水线"：

```
┌─────────────────────────────────────────────────────────────────────┐
│                         e203_core                                    │
│                                                                      │
│  ┌──────────────┐    指令(IR)      ┌──────────────────────────────┐  │
│  │   IFU        │ ──────────────► │   EXU                        │  │
│  │  (取指单元)   │                  │  (执行单元)                   │  │
│  │              │ ◄────────────── │                              │  │
│  │ • PC生成      │   跳转/异常反馈   │ • 译码(decode)               │  │
│  │ • 读ITCM      │                  │ • 读寄存器(regfile)           │  │
│  │ • 分支预测    │                  │ • 数据相关检测(disp+oitf)     │  │
│  │ • 预译码      │                  │ • 算术运算(ALU)              │  │
│  └──────────────┘                  │ • 写回寄存器(wbck)            │  │
│         │                          │ • 异常/中断处理(commit)       │  │
│         │ ICB总线                   └──────────────────────────────┘  │
│         ▼                                      │                      │
│  ┌──────────────┐                              │ ICB总线              │
│  │   ITCM       │                              ▼                      │
│  │  (指令存储)   │                     ┌──────────────┐               │
│  └──────────────┘                     │    LSU        │               │
│                                       │  (访存单元)   │               │
│  ┌──────────────┐                     │  • Load指令   │               │
│  │   DTCM       │ ◄─────────────────► │  • Store指令  │               │
│  │  (数据存储)   │                     └──────────────┘               │
│  └──────────────┘                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**为什么分成 IFU、EXU、LSU 三个单元？**

因为这三种操作在硬件上天然分离：
- **IFU**：只读指令，只访问 ITCM（或系统内存）
- **EXU**：只做计算，在寄存器堆里读写
- **LSU**：只做数据访存，访问 DTCM（或系统内存）

三者分开，可以让电路结构更清晰，复用性也更强。

## 1.3 地址空间映射

CPU 通过"地址"来区分不同的存储器和外设，就像城市里每栋房子都有门牌号。e203 的 32 位地址空间（0x0000_0000 ~ 0xFFFF_FFFF，共 4GB）按区域划分：

```
地址                     映射目标
──────────────────────────────────────────────────────────────
0x0000_0000              复位向量（MROM中的启动代码）
0x0200_0000 ~ 0x0200_FFFF  CLINT（软中断+计时器寄存器）
0x0C00_0000 ~ 0x0CFF_FFFF  PLIC（平台级中断控制器）
0x1000_0000 ~ 0x1FFF_FFFF  PPI（私有外设接口：UART/SPI/GPIO等）
0x8000_0000 ~ 0x800F_FFFF  ITCM（指令紧耦合存储器，64KB, 可配）
0x9000_0000 ~ 0x900F_FFFF  DTCM（数据紧耦合存储器，64KB, 可配）
0xF000_0000 ~ 0xFFFF_FFFF  FIO（快速I/O区域）
```

这些地址段由 `config.v` 中的宏定义确定：

```verilog
// config.v
`define E203_CFG_ITCM_ADDR_BASE   32'h8000_0000  // ITCM 从这里开始
`define E203_CFG_DTCM_ADDR_BASE   32'h9000_0000  // DTCM 从这里开始
`define E203_CFG_CLINT_ADDR_BASE  32'h0200_0000  // CLINT
`define E203_CFG_PLIC_ADDR_BASE   32'h0C00_0000  // PLIC
```

**地址路由的实现**：当 CPU 发出一个访问地址时，BIU（总线接口单元）比较地址的高位，判断发给哪个目标：

```
addr[31:28] == 4'b1000  →  送往 ITCM（0x8xxx_xxxx）
addr[31:28] == 4'b1001  →  送往 DTCM（0x9xxx_xxxx）
addr[31:28] == 4'b0001  →  送往 PPI（0x1xxx_xxxx）
...
```

## 1.4 ICB 总线协议：模块间通信的"普通话"

### 什么是 ICB？

ICB（Internal Chip Bus，片内总线）是 e203 自定义的一套模块间通信协议。所有模块之间的通信都遵循这套协议，这样每个模块只需要懂 ICB，就能和任何其他模块通信。

### ICB 的两个通道

一次访问分为两步，对应两个独立通道：

```
┌────────────────────────────────────────────────────────┐
│                  ICB 协议                               │
│                                                        │
│  [命令通道 CMD Channel]    发起方 → 接收方              │
│  ──────────────────────────────────────────────        │
│  cmd_valid : 发起方说"我有请求"                          │
│  cmd_ready : 接收方说"我准备好了"                        │
│  cmd_addr  : 访问的地址（32位）                          │
│  cmd_read  : 1=读操作, 0=写操作                         │
│  cmd_wdata : 写操作时的数据（32位）                      │
│  cmd_wmask : 写操作的字节掩码（4位，每位控制1字节）       │
│                                                        │
│  [响应通道 RSP Channel]    接收方 → 发起方              │
│  ──────────────────────────────────────────────        │
│  rsp_valid : 接收方说"响应好了"                          │
│  rsp_ready : 发起方说"我可以接收响应"                    │
│  rsp_rdata : 读操作时返回的数据（32位）                  │
│  rsp_err   : 1=访问出错（如地址非法）                    │
└────────────────────────────────────────────────────────┘
```

### 握手时序图解

```
         Cycle1   Cycle2   Cycle3   Cycle4   Cycle5
         ───────────────────────────────────────────
cmd_valid:  1        1        1                     ← 发起方有3个请求
cmd_ready:  0        1        1                     ← 接收方Cycle1还没准备好
                     ↑        ↑
                  握手#1    握手#2                  ← valid AND ready 同时为1

rsp_valid:           1        1                     ← 返回对应的响应
rsp_ready:           1        1
                     ↑        ↑
                  响应#1    响应#2
```

**关键规则**（很重要！）：
1. `valid` 一旦拉高，在握手成功前不能撤销
2. `ready` 可以在任何时候变化（不需要等 `valid`）
3. 命令通道和响应通道完全独立（可以连续发多个命令，响应稍后依次返回）

### ICB 与 AXI 的关系

熟悉 AXI 的读者会注意到 ICB 很像 AXI-lite 的一个子集。确实，ICB 是专为 e203 内部简化设计的，少了 burst、缓存类型等复杂特性，但核心握手机制相同。

### e203 中主要的 ICB 总线实例

```
总线名称                        传输内容
─────────────────────────────────────────────────────────
ifu2itcm_icb_*               IFU 取指 → ITCM
ifu2biu_icb_*                IFU 取指 → 系统内存（ITCM未命中时）
lsu2dtcm_icb_*               LSU 访存 → DTCM
lsu2biu_icb_*                LSU 访存 → 系统内存（DTCM未命中时）
ppi_icb_*                    CPU → 私有外设（UART/SPI等）
clint_icb_*                  CPU → CLINT（计时器/软中断）
plic_icb_*                   CPU → PLIC（中断控制器）
```

## 1.5 模块间的信号命名规范

理解信号名称能极大提升读代码速度。e203 采用一套一致的命名模式：

### 模式1：`发起方2接收方_总线类型_通道_信号`

```
ifu  2  itcm  _icb_  cmd  _valid
 ↑       ↑      ↑     ↑      ↑
IFU   到ITCM  ICB总线  命令通道  有效信号
```

- `2` = "to"（到）
- `icb` = ICB协议
- `cmd` = 命令通道
- `rsp` = 响应通道

### 模式2：模块内部握手 `模块名_o/i_信号`

```
ifu_o_valid   ← IFU 向 EXU 输出的指令有效信号（_o_ = output）
ifu_o_ir      ← IFU 向 EXU 输出的指令寄存器（IR）内容
disp_i_valid  ← 派发单元的输入有效（_i_ = input）
```

### 模式3：写回信号 `*_wbck_*`

```
alu_wbck_valid   ← ALU 写回有效
alu_wbck_ready   ← 写回仲裁器就绪
alu_wbck_wdat    ← 写回数据
alu_wbck_rdidx   ← 写入哪个寄存器（rd的索引）
```

## 1.6 完整的数据流：一次加法指令的全程追踪（预览）

下图是后续三章内容的"路线图"，先有个整体印象就好（每个步骤后文都会详细展开）：

```
[ITCM 存储的指令: add x3, x1, x2]
         │
         │ (1) IFU 通过 ifu2itcm_icb_cmd 发出取指请求
         ▼
[ITCM 返回 64bit 数据: ifu2itcm_icb_rsp_rdata]
         │
         │ (2) ift2icb 从 64bit 中切出正确的 32bit 指令
         ▼
[指令寄存器 IR: 0x00208133 = add x3,x1,x2]
         │
         │ (3) ifu_o_valid/ir 送给 EXU
         ▼
[decode 译码: 识别为 R型 ADD，rs1=x1, rs2=x2, rd=x3]
         │
         │ (4) 读寄存器堆
         ▼
[regfile 读出: x1=5, x2=3]
         │
         │ (5) disp 检查无数据冒险（OITF为空）→ 允许派发
         ▼
[ALU 计算: 5 + 3 = 8]
         │
         │ (6) wbck 写回
         ▼
[寄存器堆: x3 ← 8]
```
